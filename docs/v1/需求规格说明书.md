# 末日废墟生存战（AI 对抗版）需求规格说明书

## 1. 文档信息

- 文档名称：末日废墟生存战（AI 对抗版）需求规格说明书
- 文档类型：产品需求规格（PRD/SRS）
- 版本：v1.0
- 状态：评审中

## 2. 引言

### 2.1 目的
本文档用于明确《末日废墟生存战（AI 对抗版）》的业务目标、核心规则、功能需求、结算规则与验收口径，作为产品、客户端、服务端与测试协同实现的统一依据。

### 2.2 范围
本需求覆盖：
- 房间与匹配规则
- 地图与建筑规则
- 物资与状态系统
- 信息可见性与记忆规则
- 玩家动作规则
- 冲突（攻击）规则
- AI 决策规则
- 结算与终局规则

不覆盖：
- UI 视觉细节
- 网络协议字段级定义
- 具体数值平衡调优方案

### 2.3 术语定义
- 角色（Character）：本局内可行动单位，包含真人玩家角色与 AI Agent 角色。
- 玩家（Player）：默认指真人玩家；如需包含 AI，会明确写“所有角色（真人+AI）”。
- 房主（Host）：房间创建者，拥有 `start` 与房间配置权限。
- 对局（Match）：从房主 `start` 到触发终局结算结束的一场完整游戏。
- 一天（Day Cycle）：由“白天阶段 -> 夜晚阶段”组成。
- 阶段（Phase）：白天或夜晚中的一个阶段。
- 轮次（Round）：阶段内一次动作提交与统一结算单元。
- 动作（Action）：玩家在轮次内提交的单次指令（`MOVE / EXPLORE / USE / TAKE / REST / ATTACK`）；攻击结算阶段还包含后续动作（`GET / TOSS`）。
- 战利品窗口（Loot Window）：攻击判定结束后，由胜者在 `GET` 与 `TOSS` 间二选一的结算子流程。
- 本玩家阶段结束（Phase Ended for Player）：玩家死亡或执行 `REST` 后，本阶段不再行动。
- 阶段推进（Phase Advance）：当所有存活角色均阶段结束时，切换下一阶段。
- 阶段固定消耗（Base Upkeep）：每名存活角色每阶段仅扣除一次基础消耗。
- 终局模式（End Mode）：`ALL_DEAD` 或 `HUMAN_ALL_DEAD`。
- 物资快照（Resource Snapshot）：探索获得的当前建筑剩余物资记录（个人认知，默认长期保留，但可能失真/不实时）。
- 角色信息快照（Character Snapshot）：探索获得的当前建筑角色列表记录（个人认知，默认长期保留，但可能失真/不实时）。
- 探索者（Observer）：执行探索并获得快照的一方。
- 被观察方（Observed）：出现在探索者角色信息快照中的一方。

## 3. 产品概述

### 3.1 核心定位
本产品为 6 人回合制末日生存策略游戏，真人与 AI 同局对抗，唯一目标为“存活更久”。

### 3.2 核心目标
- 通过移动、探索、拿取、使用、休息、攻击等动作形成策略博弈。
- 在有限资源与高死亡风险中进行决策。
- 支持房间级终局模式以适配不同玩法节奏。

## 4. 业务规则

### 4.1 房间与匹配规则
- BR-001：单房间固定人数为 6。
- BR-002：当真人玩家数量不足 6 时，系统自动补齐 AI Agent 至 6。
- BR-003：角色构成为真人玩家（手动）+ AI Agent（自动）。
- BR-004：房间状态分为“等待中”“游戏中”。
- BR-005：仅“等待中”房间允许 `join`。
- BR-006：房主在“等待中”离开，房间解散。
- BR-007：房主在“游戏中”离开，房间关闭并立即结束该局。
- BR-008：普通玩家在“等待中”可离开并加入其他房间。
- BR-009：普通玩家在“游戏中”离开按死亡处理。
- BR-010：房主创建房间时必须指定 `end_mode`。
- BR-011：`end_mode` 支持：
  - `ALL_DEAD`：所有角色死亡时结束。
  - `HUMAN_ALL_DEAD`：所有真人玩家死亡时结束。

### 4.2 地图与区域规则
- BR-012：地图固定为 9x9 网格，坐标范围 `x=1..9, y=1..9`。
- BR-013：地图矩阵固定如下：

```text
[
  ["Q","Q","Q","Q","Q","Q","Q","Q","Q"],
  ["Q","X","W","X","X","J","X","J","Q"],
  ["Q","J","X","B","B","X","B","X","Q"],
  ["Q","X","S","M","X","W","X","J","Q"],
  ["Q","W","X","J","X","X","W","X","Q"],
  ["Q","B","X","X","X","W","J","J","Q"],
  ["Q","X","J","X","X","M","S","W","Q"],
  ["Q","X","X","X","J","B","X","B","Q"],
  ["Q","Q","Q","Q","Q","Q","Q","Q","Q"]
]
```

- BR-014：区域定义：`Q` 街道/围墙，`X` 空地，`J/B/S/W/M` 为安全区建筑。
- BR-015：进入 `Q`（昼夜）立即死亡。
- BR-016：进入 `X`（白天）立即死亡；夜晚进入不立即死亡，但显著降低夜晚存活率。
- BR-017：仅安全区可进行常规动作（探索、拿取、休息、攻击目标存在时等）。

### 4.3 物资与建筑库存规则
- BR-018：物资只能通过探索后拿取获得。
- BR-019：物资使用后立即生效，无使用次数限制；背包数量相应减少。
- BR-020：物资无过期、损坏机制。
- BR-021：建筑库存为永久固定总量，不刷新，不再生。
- BR-022：各建筑初始库存：
  - `J`：面包 x3，瓶装水 x3
  - `B`：压缩饼干 x4
  - `S`：罐头 x5，桶装水 x5
  - `W`：清水 x6
  - `M`：无可拿取物资
- BR-023：物品效果：
  - 面包：饱食 +10
  - 瓶装水：水分 +10
  - 压缩饼干：饱食 +20
  - 罐头：饱食 +20
  - 桶装水：水分 +20
  - 清水：水分 +15

### 4.4 信息可见性与记忆规则
- BR-024：玩家初到建筑时，当前建筑物资与角色信息均为未知。
- BR-025：玩家执行 `EXPLORE` 后，获得该建筑当下“物资快照 + 角色信息快照”。
- BR-026：角色信息快照为单向可见，仅探索者获知；被观察方不自动收到“已被发现”提示。
- BR-027：玩家离开建筑后，快照作为个人记忆持续保留，不做硬失效删除。
- BR-028：建筑发生相关事件后，玩家既有快照状态标记为“可能失真（STALE）”，但仍可基于该记忆继续行动。
- BR-029：玩家回到建筑时可基于记忆直接行动，也可先探索刷新认知。
- BR-029a：每轮结算后仅更新玩家个人可见信息，不提供全局实时信息。
- BR-029b：玩家会在以下“本人参与动作”后更新该建筑认知：`EXPLORE`、`TAKE`、`ATTACK`、`GET/TOSS`。
- BR-029c：`REST` 与 `USE` 不会刷新建筑物资/角色快照，仅更新自身状态或背包。

### 4.5 角色状态与死亡规则
- BR-030：角色状态包含：水分、饱食、曝光。
- BR-030-init：角色初始状态与上限（V1）：
  - 初始水分 `100`，上限 `100`
  - 初始饱食 `100`，上限 `100`
  - 初始曝光 `0`，上限 `100`
  - 初始背包：`瓶装水 x1`、`面包 x1`
- BR-030a：阶段固定消耗（Base Upkeep）：每名存活角色每阶段仅扣除一次 `水分-1、饱食-1、曝光±0`。
- BR-030b：动作额外消耗（V1）按下表执行（数值作用于“本轮动作结算”）：
  - `MOVE`：水分 `-2`，饱食 `-1`，曝光 `+2`
  - `EXPLORE`：水分 `-1`，饱食 `-1`，曝光 `+1`
  - `USE`：水分 `±0`，饱食 `±0`，曝光 `±0`（仅结算物品效果）
  - `TAKE`：水分 `-1`，饱食 `±0`，曝光 `+1`
  - `REST`：水分 `±0`，饱食 `±0`，曝光 `-3`
  - `ATTACK`：水分 `-2`，饱食 `-2`，曝光 `+3`
  - `GET`：水分 `±0`，饱食 `±0`，曝光 `±0`
  - `TOSS`：水分 `±0`，饱食 `±0`，曝光 `±0`
- BR-030c：状态边界处理：每轮结算汇总完成后，`水分/饱食/曝光` 均执行边界约束 `clamp(0,100)`；随后再执行死亡判定（`水分<=0` 或 `饱食<=0`）。
- BR-031：水分 `<=0` 或饱食 `<=0` 时立即死亡。
- BR-032：夜晚位于 `X` 时按曝光值计算存活率并判定死亡。
- BR-032a：夜晚 `X` 区存活函数（V1）：`P_survive(E) = clamp(0.97 - 0.008 * E, 0.03, 0.97)`，其中 `E` 为曝光值。
- BR-032b：夜晚 `X` 区判定方式：每轮抽样 `u∈[0,1)`；`u <= P_survive(E)` 判定存活，否则判定死亡。
- BR-032c：夜晚 `X` 区判定时机：在每轮“生死判定阶段”中，对所有当前处于 `X` 的存活角色各执行一次独立判定。
- BR-033：死亡角色背包清空，其背包剩余物品掉落至死亡所在建筑库存。
- BR-034：同轮多人死亡时，按结算顺序依次掉落。
- BR-035：同轮动作统一进入结算队列后，按“动作提交时间升序”逐个结算；先提交先结算。
- BR-036：若提交时间相同，按“进入房间顺序”结算（先进入先结算），以保证结果可复现。

### 4.6 玩家态信息与通知规则
- BR-037：所有在局玩家（包含房主）必须可见以下固定信息：
  - 我是谁（玩家标识）
  - 我所在房间（房间标识）
  - 我当前位置（坐标）
  - 当前时间态（第几天、白天/夜晚）
  - 当前建筑对我披露的信息状态（未探索/上次探索快照）
  - 当前建筑快照内容（仅本人可见口径）：角色信息与物资信息
  - 我的状态：`water/food/exposure`
  - 我的背包物品
- BR-038：操作历史仅保留最近 100 条，超出后按时间顺序滚动淘汰最旧记录。
- BR-039：每条历史/通知记录应包含最小字段：`message_id`、`server_seq`、`day`、`phase`、`round`、`timestamp`、`event_type`、`payload`。
- BR-040：通知事件可见性遵循“与本人相关才下发详情”原则；与本人无关的私有结算细节不得下发。
- BR-041：固定通知事件类型（V1）：
  - `GAME_STARTED`：通知所有人“房间N人（含AI）进入游戏”
  - `ROUND_STARTED`：通知所有人“第X天-白/夜-阶段Y，请选择操作”
  - `ACTION_ACCEPTED`：仅通知动作提交者“已选择操作，等待结算”
  - `ROUND_SETTLED`：通知所有人“本轮结算完成”，并附各自私有结算结果
- BR-042：`ROUND_SETTLED` 私有结算结果必须按动作类型输出可解释信息（消耗/增益/结果），例如：
  - `EXPLORE`：消耗与探索结果
  - `MOVE`：消耗与新坐标
  - `USE`：消耗与状态增益
  - `REST`：消耗/恢复与阶段结束状态
  - `GET`：获得物品明细
  - `ATTACK`：目标、胜负、状态变化、战利品窗口结果

## 5. 功能需求

### 5.1 回合流程
- FR-001：系统按“白天/夜晚”阶段交替运行。
- FR-002：每轮由本阶段仍可行动的存活角色提交动作并统一结算。
- FR-003：若所有存活角色均阶段结束（死亡或休息），立即推进下一阶段。
- FR-003a：单人在同一阶段可连续执行多轮动作；只要该玩家未死亡且未执行 `REST`，阶段不得提前结束。

### 5.2 玩家动作
- FR-004：每名存活玩家每轮最多提交 1 个核心动作。
- FR-005：核心动作类型包含 `MOVE / EXPLORE / USE / TAKE / REST / ATTACK`；攻击后续动作包含 `GET / TOSS`。
- FR-006：`MOVE` 仅允许上下左右相邻 1 格，且目标坐标不得超出地图边界（`x=1..9, y=1..9`）；可进入任意地块，进入 `Q/X` 后按生死规则结算。
- FR-007：`EXPLORE` 仅安全区可执行，产出物资快照与角色信息快照。
- FR-008：`USE(items)` 消耗背包物品，恢复对应状态值。
- FR-009：`TAKE(items)` 在已探索建筑执行，每轮最多拿取 3 件；若记忆与真实库存不一致，可能出现拿取失败或部分成功。
- FR-010：`REST` 降低曝光，并使该玩家本阶段结束。
- FR-011：`ATTACK(target)` 仅可针对同建筑且“已被自己发现”的目标。
- FR-012：真人玩家超时未提交动作时，系统自动提交 `REST`。
- 说明（后续动作）：`GET / TOSS` 不属于本轮“核心动作 6 选 1”，仅在攻击判定后的战利品窗口触发。
- `GET(items)`：攻击胜者从败者背包中选择最多 3 件物品拿取。
- `TOSS`：攻击胜者放弃本次拿取，直接进入后续生死判定。
- 超时处理：战利品窗口超时未选择时，系统默认执行 `TOSS`。
- 详细规则见 5.3（FR-019 ~ FR-022）。

### 5.3 攻击与冲突规则
- FR-013：攻击胜负依据“攻击值”比较判定。
- FR-014：攻击值公式：`基础攻击值 + 状态修正 + 信息优势加成`。
- FR-015：信息优势加成规则：
  - 若 `A发现B` 且 `B未发现A`，A 获得 `+2`。
  - 若 A 与 B 互相发现，双方均 `+2`（效果等效抵消）。
- FR-016：信息优势加成在 V1 固定为 `+2`。
- FR-017：攻击判定后，胜者（可为发起方或反杀方）可从败者背包夺取最多 3 件物品。
- FR-018：若攻击引发死亡，结算顺序必须为“胜者夺取 -> 生死判定 -> 死亡掉落”。
- FR-019：攻击判定后必须触发战利品窗口，胜者二选一执行后续动作：`GET(items)` 或 `TOSS`。
- FR-020：`GET(items)` 仅攻击胜者可执行，且单次最多选择 3 件败者背包物品。
- FR-021：`TOSS` 仅攻击胜者可执行，表示放弃本次夺取，直接进入后续生死判定。
- FR-022：若战利品窗口超时未选择，系统默认执行 `TOSS`。
- 规则参数（V1，固定值）：
  - 攻击值总公式：`AttackScore = Base + StateMod + InfoMod + Rand`
  - `Base = 10`
  - `InfoMod`：沿用 FR-015，信息优势单边 `+2`，互相发现时双方均 `+2`（等效抵消）
  - `Rand`：离散随机项，取值集合 `{-1, 0, +1}`
  - `StateMod = ResourceMod + ExposureMod`
  - `ResourceMod`：令 `R = (水分 + 饱食) / 2`
    - `R >= 80` -> `+3`
    - `60 <= R < 80` -> `+2`
    - `40 <= R < 60` -> `0`
    - `20 <= R < 40` -> `-2`
    - `R < 20` -> `-4`
  - `ExposureMod`：
    - `曝光 < 20` -> `+1`
    - `20 <= 曝光 < 60` -> `0`
    - `60 <= 曝光 < 80` -> `-1`
    - `曝光 >= 80` -> `-2`
  - 胜负阈值：令 `Δ = AttackScore(发起方) - AttackScore(被攻击方)`
    - `Δ >= 2`：发起方胜
    - `Δ <= -2`：被攻击方胜（反杀）
    - `-1 <= Δ <= 1`：僵持（无人胜）
  - 胜负强度分级（用于后续状态扣减幅度映射）：
    - `|Δ| = 2~3`：小胜
    - `|Δ| = 4~5`：中胜
    - `|Δ| >= 6`：大胜

### 5.4 AI 行为规则
- FR-023：AI 无操作超时，由系统自动决策并提交动作。
- FR-024：AI 目标为“最大化生存概率”。
- FR-025：AI 不进入白天 `Q/X`，不执行会导致水分/饱食归零的动作。
- FR-026：AI 白天优先探索/移动/收集，夜晚优先休息/隐蔽。
- FR-027：资源不足时，AI 优先探索、拿取、使用物资。
- FR-028：AI 仅在濒临死亡且状态优势明显时倾向攻击，否则优先撤离。
- FR-028a：AI 策略层必须采用统一接口：`choose_action(obs, action_mask) -> action`。
- FR-028b：V1 支持 `RuleBot`、`LLMPolicy` 两类策略实现，且均遵循同一动作合法性校验。
- FR-028c：`LLMPolicy` 必须采用两层机制：发现层（技能索引加载）+ 意图层（技能匹配、条件校验、详情加载），并在超时/失败时回退 `RuleBot`。
- FR-028d：V1 必须支持“场景模板路由”机制，根据昼夜阶段、战利品窗口、生存风险等上下文选择策略模板。
- FR-028e：策略模板需支持配置化管理（可新增/修改/停用），且不影响策略接口与结算主流程。
- FR-028f：策略模板执行前后均需经过 `action_mask` 与参数合法性校验；不合法动作不得进入结算队列。
- FR-028g：策略模板索引必须采用标准元信息结构（至少包含 `skill_id/name/short_desc/version/status/cost_tag`），并支持版本管理与兼容校验。

### 5.5 玩家信息与消息功能
- FR-029：客户端每轮渲染时必须提供 BR-037 定义的全部固定信息。
- FR-030：客户端必须维护最近 100 条操作历史并滚动更新（按 `server_seq` 单调展示）。
- FR-031：系统必须按 BR-041 触发并下发通知事件；`ACTION_ACCEPTED` 仅下发给提交者。
- FR-032：`ROUND_SETTLED` 必须同时包含公共结算完成信号与“仅本人可见”的私有结果明细。
- FR-033：断线重连后，系统应返回最近 100 条历史，客户端按 `server_seq` 去重并补齐。
- FR-034：非法操作或失败操作必须返回失败通知，包含失败原因与当前可执行动作提示。

### 5.6 AI 工程约束（V1）
- FR-035：`choose_action` 输出必须满足严格 JSON Schema（固定字段、类型、必填项、动作参数 oneOf），且禁止未声明字段。
- FR-036：当多个策略模板同时命中时，必须按确定性规则决议：`priority` > 条件匹配分 > 固定兜底顺序。
- FR-037：LLM Prompt 必须采用固定分段组装：`System 规则` + `Skill 模板` + `State JSON` + `Constraint(action_mask/schema)`。
- FR-038：系统必须具备 OpenAI 调用运行策略：限流、超时、熔断、错误分类回退。
- FR-039：LLM 调用失败（超时、429、5xx、解析失败、Schema 不合法）时，必须回退 `RuleBot`。

## 6. 结算需求

### 6.1 常规结算
- SR-001：执行每阶段一次基础固定消耗。
- SR-002：叠加本轮动作额外消耗（按 BR-030b，覆盖 `MOVE/EXPLORE/USE/TAKE/REST/ATTACK/GET/TOSS`）。
- SR-003：结算使用、拿取、探索、休息、攻击等动作结果。
- SR-003a：同轮中共享资源冲突（如多人 `TAKE` 同一物资）按结算队列顺序执行，后结算方可能因库存变化拿取失败。
- SR-003b：本轮状态更新完成后，先执行状态边界处理（按 BR-030c），再进入生死判定。
- SR-004：攻击场景先进行攻击判定，再进入战利品窗口（`GET/TOSS`），之后执行生死判定与掉落。
- SR-005：完成生死判定并移除死亡角色。
- SR-006：结算结束后切换时段并进入下一轮。

### 6.2 全员休息特殊推进
- SR-007：当本轮所有角色均为 `REST`（含超时自动休息）时，触发特殊推进。
- SR-008：若该阶段基础消耗尚未扣除，仅扣一次基础消耗（水分 -1、饱食 -1）。
- SR-009：跳过其余常规动作结算，直接进入下一轮并切换时段。

### 6.3 终局结算
- SR-010：每轮结算后按房间 `end_mode` 检查终局条件。
- SR-011：`ALL_DEAD`：总存活数为 0 时立即终局。
- SR-012：`HUMAN_ALL_DEAD`：真人存活数为 0 时立即终局。
- SR-013：终局需产出：
  - 角色统计：存活轮次、获取物资总量、死亡原因
  - 真人战绩：最后存活真人、真人存活时长、真人击杀/被击杀
  - 房间排名：按存活轮次降序
- SR-014：终局后重置房间数据，支持下一局。

### 6.4 通知结算
- SR-015：`GAME_STARTED` 在对局初始化完成后广播一次。
- SR-016：每轮进入可操作状态时广播 `ROUND_STARTED`。
- SR-017：单个玩家提交动作后立即向其发送 `ACTION_ACCEPTED`。
- SR-018：所有应结算动作处理完成后广播 `ROUND_SETTLED`；公共部分一致，私有详情按玩家隔离下发。
- SR-019：历史写入顺序必须与 `server_seq` 一致，且遵循 100 条滚动窗口策略。

## 7. 配置项需求

- CFG-001：`end_mode`（必配）：`ALL_DEAD | HUMAN_ALL_DEAD`
- CFG-002：`round_action_timeout_sec`（必配）：`90`
  - 含义：`ROUND_STARTED` 广播后，核心动作提交超时时间（秒）。
  - 处理：真人玩家超时自动 `REST`；AI 由系统自动决策并提交动作。
- CFG-003：`loot_window_timeout_sec`（必配）：`60`
  - 含义：攻击后 `GET/TOSS` 战利品窗口超时时间（秒）。
  - 处理：超时默认 `TOSS`。

## 8. 非功能与约束

- NFR-001：结算流程必须服务端原子化执行，避免并发导致状态不一致。
- NFR-002：所有规则判定以服务端状态为准，客户端仅展示可见信息。
- NFR-003：信息可见性严格遵循“个人认知模型”，不得泄露全局实时角色/库存。
- NFR-004：通知下发必须具备幂等消费能力（基于 `message_id` 或 `server_seq` 去重）。
- NFR-005：私有结算结果必须进行用户隔离，禁止跨玩家泄露。

## 9. 验收要点（UAT）

- AC-001：玩家未探索时无法获知建筑角色列表。
- AC-002：A 发现 B 且 B 未发现 A 时，A 攻击值确实获得 `+2`。
- AC-003：A/B 互相发现时，双方加成等效抵消。
- AC-004：攻击胜者可夺取败者最多 3 件物品。
- AC-005：攻击致死时，先夺取后掉落，顺序正确。
- AC-006：全员休息场景仅执行一次阶段基础消耗并快速推进。
- AC-007：两种终局模式均能在对应条件下立即触发。
- AC-008：攻击判定后必须出现 `GET/TOSS` 二选一流程。
- AC-009：攻击战利品窗口超时时，系统默认执行 `TOSS` 且流程继续。
- AC-010：同轮 A 先提交 `TAKE`、B 后提交 `TAKE` 且目标为同一库存时，A 先结算成功后，B 会因库存不足而失败（或部分成功）。
- AC-011：攻击值按 `AttackScore = Base + StateMod + InfoMod + Rand` 计算，且 `Base=10`、`Rand∈{-1,0,+1}`。
- AC-012：同状态下，单边信息优势（发现关系单向成立）会稳定带来 `+2` 攻击值差异。
- AC-013：`Δ` 在 `[-1,1]` 时判定僵持，`Δ>=2` 与 `Δ<=-2` 分别稳定判定发起方胜/被攻击方胜。
- AC-014：夜晚位于 `X` 时，曝光 `E=20` 的存活率按函数计算为 `0.81`。
- AC-015：夜晚 `X` 区存活率函数在全域受 `clamp` 约束，结果始终满足 `0.03 <= P_survive <= 0.97`。
- AC-016：玩家执行 `REST` 时不会获得建筑库存变化信息；若他人在同建筑拿取，玩家认知保持旧快照（可能失真）。
- AC-017：玩家执行 `EXPLORE` 后，可获得当前建筑最新物资与角色快照并覆盖旧认知。
- AC-018：动作额外消耗严格按 BR-030b 结算；例如 `MOVE` 必须产生 `水分-2、饱食-1、曝光+2`。
- AC-019：`USE/GET/TOSS` 不产生动作额外消耗，仅结算其业务效果（物品增益或放弃拿取）。
- AC-020：新角色初始属性必须为 `水分100/饱食100/曝光0`，且初始背包为 `瓶装水x1 + 面包x1`。
- AC-021：在单人存活场景中，玩家不执行 `REST` 且未死亡时，系统必须允许同阶段继续进入下一轮动作。
- AC-022：任意轮结算后，角色状态值必须满足 `0<=水分/饱食/曝光<=100`，且死亡判定在边界约束之后执行。
- AC-023：夜晚每一轮生死判定阶段中，所有处于 `X` 的存活角色均应各触发一次独立存活率判定。
- AC-024：所有在局玩家（含房主）均可看到 BR-037 定义的固定信息字段。
- AC-025：操作历史严格保留最近 100 条，新增第 101 条时最旧一条被淘汰。
- AC-026：每轮必须按顺序触发：`ROUND_STARTED` -> `ACTION_ACCEPTED`（对提交者）-> `ROUND_SETTLED`。
- AC-027：`ROUND_SETTLED` 中不得包含与接收玩家无关的私有结算细节。
- AC-028：`choose_action` 返回含未声明字段或字段类型不合法时，动作不得入队并回退 `RuleBot`。
- AC-029：同一输入下模板冲突决议结果必须稳定可复现（决议规则一致）。
- AC-030：Prompt 组装必须遵循固定四段结构，且不得混入未授权字段。
- AC-031：Discovery 或 Intent 任一阶段超时后，系统必须在本轮预算内回退 `RuleBot` 并继续流程。
- AC-032：OpenAI 返回 `429/5xx`、解析失败、Schema 校验失败时，均应触发 `RuleBot` 回退并记录失败原因。
- AC-033：`action_mask` 禁止动作即使由 LLM 输出，也不得进入结算队列。
