## 2. 设计目标与原则

### 2.1 设计目标

- 以服务端权威模型实现规则一致性与可复现结算。
- 支持真人+AI 混合 6 人对局，满足回合制与昼夜阶段推进规则。
- 完整落地“玩家信息面板 + 通知机制”新增需求。
- 在 2C2G 无 GPU 服务器条件下可稳定运行线上推理。
- 保持工程可维护性与可观测性，支撑当前阶段稳定交付。

### 2.2 关键设计原则

- 权威性：所有规则判定、随机抽样、结算顺序由服务端执行。
- 隐私隔离：私有结算与认知快照严格按玩家视角过滤。
- 确定性：排序规则固定（提交时间、入房顺序），随机源可复盘。
- 幂等性：通知和历史消息以 `message_id/server_seq` 去重消费。
- 降级优先：AI 超时或模型异常时自动切换规则 Bot，保障局不中断。

## 3. 总体架构

### 3.1 架构概览

采用“全 Python 前后端 + 服务端权威结算”方案：

- 客户端：NiceGUI（Python）渲染网页。
- 接入层：FastAPI（HTTP + WebSocket）。
- 领域层：游戏规则引擎（房间、阶段、轮次、动作结算）。
- 事件层：通知服务（事件构建、可见性过滤、历史滚动、重连补齐）。
- AI 层：RuleBot + LLMPolicy（统一策略接口，含回退）。
- 存储层：V1 内存态（房间状态、会话与消息历史）。

### 3.2 逻辑分层

1. `api`：HTTP 接口、WS 连接管理、请求鉴权。
2. `application`：用例编排（创建房间、开始对局、提交动作、重连恢复）。
3. `domain`：核心实体与规则（Room、Player、Round、Action、Snapshot）。
4. `engine`：结算管线（排序、执行、状态钳制、生死判定、终局检查）。
5. `notification`：事件定义、消息序列、私有/公共分发、历史窗口。
6. `ai`：策略接口、LLM 策略器、规则回退器。
7. `infra`：内存仓储、日志、配置、可观测性。

## 4. 运行模式与资源预算

### 4.1 运行模式

- 策略决策：`RuleBot + LLMPolicy`。
- LLM 调用：通过 OpenAI API 在线推理。
- 稳定性：任一层超时/失败时回退 `RuleBot`。

### 4.2 2C2G 服务器预算策略

- 推理：按轮次触发 LLM 决策，不做高频实时推理。
- 超时：按两层配置执行（`discovery=600ms`、`intent=1200ms`、总预算 `2000ms`），失败回退 `RuleBot`。
- 并发建议：先压测 20~50 活跃房间，依据 p95 延迟逐步放量。

## 6. 回合与结算设计

### 6.1 回合状态机

1. `ROUND_OPEN`：广播 `ROUND_STARTED`，接收动作。
2. `ROUND_LOCK`：达到提交条件或超时，冻结动作队列。
3. `ROUND_SETTLING`：按排序规则逐项结算。
4. `ROUND_BROADCAST`：广播 `ROUND_SETTLED`（公共+私有）。
5. `PHASE_CHECK`：检查阶段推进条件。
6. `GAME_OVER_CHECK`：检查终局条件。

### 6.2 排序与一致性

- 主排序：`server_received_at` 升序。
- 次排序：`join_seq` 升序。
- 排序结果写入轮次日志，供回放和争议排查。

### 6.3 结算管线

1. 阶段基础消耗（每阶段一次）。
2. 动作额外消耗与动作效果。
3. 攻击判定与战利品窗口。
4. 状态边界处理 `clamp(0,100)`。
5. 生死判定（含夜晚 X 区概率判定）。
6. 死亡掉落。
7. 阶段推进与终局检查。

## 7. 玩家信息面板设计

### 7.1 服务端投影视图（Player View）

按 BR-037 输出固定字段，面向单个玩家投影：

- `identity`: `player_id`, `room_id`
- `time_state`: `day`, `phase`, `round`
- `position`: `x`, `y`, `tile_type`
- `building_info_state`: `UNEXPLORED|SNAPSHOT|STALE`
- `building_snapshot`:
  - `resources`（当前认知）
  - `characters`（当前认知）
  - `snapshot_updated_at`
- `self_status`: `water`, `food`, `exposure`
- `inventory`
- `allowed_actions`

### 7.2 客户端渲染要求

- 每轮按最新 `state_snapshot` 全量刷新面板。
- 行动按钮由 `allowed_actions` 控制可点击状态。
- 面板中“建筑信息状态”需显式展示 `STALE` 标记。

## 8. 通知与历史机制设计

### 8.1 事件定义

统一消息模型：

- `message_id`: UUID
- `server_seq`: 房间内全局单调递增
- `day`, `phase`, `round`
- `timestamp`
- `event_type`
- `payload`

固定事件类型（V1）：

- `GAME_STARTED`
- `ROUND_STARTED`
- `ACTION_ACCEPTED`
- `ROUND_SETTLED`
- `ACTION_REJECTED`（用于 FR-034 非法/失败动作反馈）

### 8.2 可见性策略

- 公共事件：所有在局玩家可见（如 `ROUND_STARTED`）。
- 私有事件：仅目标玩家可见（如 `ACTION_ACCEPTED`）。
- 混合事件：`ROUND_SETTLED` 公共部分一致，`private_result` 按玩家单独构造。

### 8.3 历史窗口与重连补齐

- 每玩家历史仅保留最近 100 条。
- 新消息写入后超限即淘汰最旧项。
- 重连接口按 `server_seq > last_seen_seq` 回补；客户端二次去重。

### 8.4 事件时序

每轮标准顺序：

1. `ROUND_STARTED`
2. `ACTION_ACCEPTED`（每个提交者单独触发）
3. `ROUND_SETTLED`

## 9. AI 设计（RuleBot + LLMPolicy）

### 9.1 设计边界

- V1 采用统一策略接口：`choose_action(obs, action_mask) -> action`。
- V1 策略实现为 `RuleBot` 与 `LLMPolicy`，均经过同一合法性校验。
- 任一策略失败时必须回退 `RuleBot`，保证对局连续性。

### 9.2 在线决策流程

1. 构建 AI 观测（遵循个人认知，不泄露全局真值）。
2. 生成 `invalid action mask`。
3. 场景模板路由选择策略模板（如白天探索、夜晚保命、战利品窗口）。
4. 执行策略（`LLMPolicy` 或 `RuleBot`）得到候选动作。
5. 输出动作并通过规则引擎二次校验。
6. 失败或超时回退 RuleBot。

### 9.3 RuleBot 回退策略

- 优先保证存活：低资源优先补给与探索。
- 夜晚风险规避：倾向 `REST` 或撤离高风险动作。
- 无法确定安全动作时默认 `REST`。

### 9.4 AI 决策输入字段清单（V1）

#### 9.4.1 策略输入特征（Observation Schema，V1 精简版）

- `self_state`：自身生存状态向量。
  - `water`（归一化到 `[0,1]`，含义：当前水分占上限比例）
  - `food`（归一化到 `[0,1]`，含义：当前饱食占上限比例）
  - `exposure`（归一化到 `[0,1]`，含义：当前曝光占上限比例）
  - `alive`（`0/1`，含义：是否存活）
  - `phase_ended`（`0/1`，含义：本阶段是否已结束行动）
- `self_position`：自身空间位置与当前地块类型。
  - `x`（归一化到 `[0,1]`，含义：横坐标）
  - `y`（归一化到 `[0,1]`，含义：纵坐标）
  - `tile_type_onehot`（`Q/X/J/B/S/W/M`，含义：当前脚下地块类别）
- `time_state`：当前时间推进状态。
  - `day_norm`（含义：第几天的归一化表示）
  - `phase_onehot`（`DAY/NIGHT`，含义：白天或夜晚）
  - `round_norm`（含义：当前阶段轮次归一化表示）
- `inventory_vector`：自身背包资源向量（均按上限归一化）。
  - `bread`（含义：面包数量）
  - `bottled_water`（含义：瓶装水数量）
  - `compressed_biscuit`（含义：压缩饼干数量）
  - `canned_food`（含义：罐头数量）
  - `barrel_water`（含义：桶装水数量）
  - `clean_water`（含义：清水数量）
- `current_building_memory`：当前建筑的个人认知快照。
  - `info_state_onehot`（`UNEXPLORED/SNAPSHOT/STALE`，含义：认知状态）
  - `resource_snapshot_vector`（按物品类型计数归一化，含义：记忆中的库存）
  - `character_snapshot_count`（归一化，含义：记忆中的角色数量）
- `local_map_current_3x3`：以自己为中心的当前 3x3 局部信息（直接可确定信息）。
  - 对每个格子编码：`tile_type_onehot` + `in_map_flag`
  - 中心格额外包含：`self_here=1`
  - 含义：提供局部地形先验，支持移动/风险决策。
- `action_mask`：动作合法性掩码。
  - `can_move_up/down/left/right`（含义：对应方向是否可提交）
  - `can_explore`（含义：当前是否可探索）
  - `can_use`（含义：当前是否可使用物品）
  - `can_take`（含义：当前是否可拿取物资）
  - `can_rest`（含义：当前是否可休息）
  - `can_attack`（含义：当前是否可攻击）
  - `can_get`（含义：当前是否处于可 GET 战利品窗口）
  - `can_toss`（含义：当前是否处于可 TOSS 战利品窗口）

说明：
- 策略仅使用“玩家可见信息 + 个人记忆信息”，不输入全局真值。
- V1 仅使用上述精简字段，优先保证决策稳定性与上线效率。

#### 9.4.2 动作与参数编码（Action Schema）

- `action_type`（离散）：`MOVE|EXPLORE|USE|TAKE|REST|ATTACK|GET|TOSS`
- `MOVE`
  - `move_dir`：`UP|DOWN|LEFT|RIGHT`
- `EXPLORE`
  - 无参数
- `USE`
  - `items[]`：`[{item_type, qty}]`，`qty >= 1`
- `TAKE`
  - `items[]`：`[{item_type, qty}]`
  - 约束：单次总件数 `<= 3`
- `REST`
  - 无参数
- `ATTACK`
  - `target_player_id`（必须同建筑且已被发现）
- `GET`
  - `items[]`：`[{item_type, qty}]`
  - 约束：单次总件数 `<= 3`，仅胜者可选
- `TOSS`
  - 无参数

说明：
- 模型输出动作后必须经过规则引擎合法性校验。
- 非法动作由 `action_mask` 抑制；若仍出现非法动作，服务端拒绝并回退 RuleBot。

#### 9.4.3 字段含义说明

- Observation Schema（输入特征）
  - 含义：AI 在当前决策时刻“能看到/能记得”的信息字段集合。
  - 作用：决定模型是否具备正确决策上下文。
- Action Schema（动作参数编码）
  - 含义：AI 输出动作的标准格式（动作类型 + 必要参数）。
  - 作用：保证“模型输出 -> 规则引擎执行”可直接映射。
- `action_mask`
  - 含义：当前状态下各动作是否可执行的合法性掩码。
  - 作用：在采样前屏蔽非法动作，降低无效决策与拒绝率。
- `UNEXPLORED / SNAPSHOT / STALE`
  - 含义：建筑信息认知状态，分别表示“未探索 / 已刷新快照 / 快照可能过期”。
  - 作用：约束 AI 仅基于个人认知决策，符合信息不完全规则。
- `local_map_current_3x3`
  - 含义：以自己为中心的当前局部可确定信息。
  - 作用：在不泄露全局真值前提下，增强移动与探索决策能力。

### 9.5 LLMPolicy 两层机制（OpenAI）

#### 9.5.1 目标

- 在满足决策质量前提下，优先降低 token 与时延。
- 先根据场景选技能，再按需加载技能详情，避免全量上下文展开。
- 失败场景统一回退 `RuleBot`，保证对局连续性。

#### 9.5.2 两层定义

- 第一层：发现层（Discovery Layer）
  - 输入：技能索引（`skill_id/name/icon/short_desc/cost_tag`）+ 当前场景标签（如 `DAY/NIGHT/LOOT_WINDOW`）。
  - 目的：快速确定候选技能集合，不加载技能完整细节。
- 第二层：意图层（Intent Layer）
  - 目的：在候选技能中完成匹配与决策。
  - 任务：
    1. 自然语言意图转 `skill_id`
    2. 复杂策略决策（候选技能排序）
    3. 动态条件判断（阶段、资源、目标、冷却/窗口约束）
  - 加载：仅对选中技能按需加载完整详情，不做全量加载。

说明：
- 两层均禁止输入全局真值与他人私有信息。
- 执行结果处理作为隐式阶段，不单独对外暴露层级。

#### 9.5.3 超时与回退策略

1. 发现层超时/失败：直接回退 `RuleBot`。
2. 意图层超时/失败：直接回退 `RuleBot`。
3. 任意层输出非法或空动作：不入队，直接回退 `RuleBot`。

校验规则：
- 输出必须满足 Action Schema。
- 参数必须满足业务约束（如 `TAKE/GET` 总件数 `<= 3`，`ATTACK` 目标合法）。
- 不满足则视为失败，不进入结算队列。

#### 9.5.4 Prompt 与输出约束

- 使用固定 System Prompt，明确：
  - 只可返回 JSON，不得返回解释文本。
  - 仅可从 `action_mask` 允许动作中选择。
  - 若不确定，优先保守动作（通常为 `REST` 或安全补给动作）。
- 输出 JSON 结构：
  - `action_type`
  - `payload`
  - `confidence`（`0~1`，用于决策置信度观测）
  - `reason_code`（枚举，如 `SURVIVAL_FIRST`、`RESOURCE_RECOVER`、`ATTACK_OPPORTUNITY`）

#### 9.5.5 建议配置（V1）

- `llm_policy_enabled = true`
- `llm_mode = "discovery_intent"`
- `llm_discovery_timeout_ms = 600`
- `llm_intent_timeout_ms = 1200`
- `llm_total_budget_ms = 2000`
- `llm_fallback_policy = "RuleBot"`

#### 9.5.6 监控指标

- `llm_discovery_success_rate`
- `llm_intent_success_rate`
- `llm_skill_match_failure_rate`
- `llm_invalid_action_rate`
- `llm_timeout_rate`
- `llm_fallback_rate_to_rulebot`

### 9.6 策略模板机制（Skill-like）

#### 9.6.1 设计目标

- 参考 skill 的“先路由、后执行”思想，降低 token 成本并提升决策聚焦度。
- 在不改变策略接口前提下，支持模板化扩展和场景化决策。

#### 9.6.2 两层结构

- 场景模板层（模板路由）
  - 按状态选择模板，不直接做动作输出。
  - 默认模板建议：
    - `day_explore_collect`：白天探索/收集优先
    - `night_survival_control`：夜晚风险控制与休息优先
    - `loot_window_decision`：`GET/TOSS` 窗口决策
    - `emergency_survival`：濒死保命决策
- 决策层（Discovery/Intent）
  - 发现层：只看技能索引，快速选候选。
  - 意图层：按需加载技能详情并完成技能匹配与条件校验。

#### 9.6.3 执行顺序

1. 规则引擎先生成 `action_mask`（硬约束）。
2. 模板路由选择 `skill_id`。
3. 在该模板下按 Discovery/Intent 调用 LLMPolicy。
4. 输出动作通过 Action Schema 与业务规则校验。
5. 失败或超时时直接回退 RuleBot。

#### 9.6.4 模板配置模型（建议）

- `skill_id`：模板唯一标识。
- `enabled`：是否启用。
- `priority`：模板优先级。
- `match_conditions`：触发条件（如 `phase=DAY`、`loot_window=true`、`water<0.2`）。
- `intent_prompt_template`：意图层自然语言模板（V1 采用内联模板）。
- `discovery_fields`：发现层字段清单（技能索引字段）。
- `intent_fields`：意图层字段清单（匹配/校验所需字段）。
- `timeout_ms`：模板级调用超时覆盖配置（可选）。
- `fallback`：失败回退策略（默认 `RuleBot`）。

说明：
- 模板配置建议使用 Markdown + YAML Front Matter（`.skill.md`），支持热更新或重启加载。
- 新增模板应为“配置变更优先”，避免改动核心结算代码。
- V1 不强制 `prompt_template_ref`；如需外置模板，作为后续扩展能力。

#### 9.6.5 技能索引规范（参考 Claude Skills 思路）

为保证模板可维护、可检索、可扩展，技能索引建议采用“元信息 + 能力声明 + 契约”三段结构。

- `metadata`（元信息）
  - `skill_id`：唯一标识（建议 kebab-case）。
  - `name`：展示名。
  - `short_desc`：一句话用途描述（用于 Discovery）。
  - `version`：语义化版本（如 `1.0.0`）。
  - `owner`：维护人/模块。
  - `status`：`active|disabled|deprecated`。
- `capability`（能力声明）
  - `phase_scope`：适用阶段（`DAY|NIGHT|LOOT_WINDOW|ANY`）。
  - `goal_tags`：策略目标标签（如 `survival/resource/combat`）。
  - `cost_tag`：成本等级（`low|medium|high`，用于预算控制）。
  - `risk_level`：风险等级（`safe|normal|aggressive`）。
  - `priority`：路由优先级。
- `contract`（输入输出契约）
  - `discovery_fields`：发现层允许读取字段白名单。
  - `intent_fields`：意图层允许读取字段白名单。
  - `output_schema_ref`：输出 JSON Schema 引用。
  - `guardrails`：硬约束声明（必须经过 `action_mask` 与业务校验）。
  - `timeouts`：`discovery_timeout_ms`、`intent_timeout_ms`。
  - `fallback`：失败回退策略（固定 `RuleBot`）。

推荐附加规范：
- 向后兼容：同 `skill_id` 的非破坏性更新仅允许增加可选字段。
- 破坏性变更：必须提升主版本号，并提供迁移说明。
- 观测要求：每次执行记录 `skill_id/version/latency/result/fallback`。

### 9.7 `choose_action` 严格 Schema 规范

- 顶层字段固定：
  - `action_type`（必填，enum）
  - `payload`（必填，按动作类型 oneOf）
  - `confidence`（可选，`0~1`）
  - `reason_code`（可选，enum）
- 校验规则：
  - `additionalProperties=false`
  - 不允许缺失必填字段
  - `payload` 必须匹配对应动作分支参数
  - `TAKE/GET` 需满足件数约束（Schema + 业务规则双重校验）
- 处理规则：
  - 校验失败动作不入队
  - 立即回退 `RuleBot`
  - 记录失败类型（schema/type/missing-field/extra-field）

### 9.8 模板路由冲突决议

- 决议顺序：
  1. `priority`（高优先）
  2. 条件匹配分（更具体条件优先）
  3. 固定兜底顺序（建议：`loot_window_decision > emergency_survival > night_survival_control > day_explore_collect`）
- 要求：
  - 同输入必须得到同一 `skill_id`
  - 决议过程需写入调试日志，支持复盘

### 9.9 Prompt 组装规范

- 固定四段：
  1. `System`：全局规则与硬约束
  2. `Skill`：模板策略说明
  3. `State`：结构化状态 JSON
  4. `Constraint`：`action_mask` 与输出 Schema 摘要
- 安全要求：
  - 禁止直接拼接未清洗自由文本到 System 段
  - 所有动态文本需长度截断与字符清洗
  - 仅允许白名单字段进入 Prompt

### 9.10 OpenAI 调用运行策略（V1）

- 限流：
  - 房间级并发限制（防止同房间风暴）
  - 全局并发上限（防止全局雪崩）
- 超时：
  - `discovery_timeout_ms=600`
  - `intent_timeout_ms=1200`
  - `total_budget_ms=2000`
- 熔断：
  - 当超时率/错误率超过阈值时，临时切换全量 `RuleBot`
- 错误分类回退：
  - `timeout`
  - `429`
  - `5xx`
  - `parse_error`
  - `schema_invalid`
  - 以上均直接回退 `RuleBot`
- 观测字段：
  - `request_id`
  - `skill_id`
  - `latency_ms`
  - `token_usage`
  - `fallback_reason`

## 10. API 与 WebSocket 设计

### 10.1 HTTP API（V1）

- `POST /api/rooms`
  - 创建房间，参数：`end_mode`
- `POST /api/rooms/{room_id}/join`
  - 加入房间
- `POST /api/rooms/{room_id}/start`
  - 房主开局
- `GET /api/rooms/{room_id}/replay/messages?after_seq={n}`
  - 重连补历史

### 10.2 WebSocket 事件

客户端 -> 服务端：

- `submit_action`
  - `action_type`, `payload`, `day`, `phase`, `round`

服务端 -> 客户端：

- `state_snapshot`
- `event_message`（通知统一载体）
- `error`（协议级错误）

### 10.3 协议约束

- 服务端校验 `day/phase/round` 与当前局一致，不一致直接拒绝。
- 所有动作提交均返回受理或拒绝消息。
- 事件投递至少一次，客户端按 `message_id/server_seq` 幂等消费。

### 10.4 超时配置（V1）

- `round_action_timeout_sec = 90`
  - 含义：`ROUND_STARTED` 广播后，核心动作提交超时时间。
  - 超时处理：真人玩家自动提交 `REST`；AI 由系统决策并提交动作。
- `loot_window_timeout_sec = 60`
  - 含义：攻击判定后战利品窗口（`GET/TOSS`）的超时时间。
  - 超时处理：系统默认执行 `TOSS`。

## 11. 数据持久化设计（V1）

### 11.1 V1（快速上线）

- 房间状态：进程内内存存储。
- 消息历史：进程内环形缓冲区（每玩家 100 条）。
- 风险：服务重启后状态丢失。

## 12. 安全与一致性

- 服务端原子化结算（单房间串行执行）。

- 私有信息过滤器在发送前最后一步执行。
- 关键日志记录：动作入队、排序结果、结算摘要、消息序列。

## 17. 风险与缓解

- 风险：单进程内存态在重启后丢局。
  - 缓解：定期进程内快照与关键日志落盘，支持人工恢复。
- 风险：2C2G 下峰值结算超时。
  - 缓解：限房、批量推理、AI 超时回退。
- 风险：私有信息泄露。
  - 缓解：统一可见性过滤器 + 专项测试 + 审计日志。
- 风险：模型策略不稳定。
  - 缓解：灰度发布、A/B 对照、可回滚 RuleBot。
