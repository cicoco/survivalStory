# 游戏页通信与战利品流程优化方案

## 1. 目标

本方案用于统一以下改造方向：

- 攻击与战利品流程语义清晰化（同一回合内子流程）
- WebSocket 与 HTTP 职责边界清晰化
- `/view` 请求频率优化（减少不必要刷新）
- `/history` 使用策略收敛（保留补偿价值，减少状态耦合）

---

## 2. 现状（As-Is）

## 2.1 通信现状

1. WebSocket（改造前）
- 使用 `/ws/{room_id}/{player_id}` 实时接收事件。
- 事件包含 `day/phase/round`，但 `ROUND_STARTED` 不带 `round_timer`。
- 战利品窗口复用 `ROUND_STARTED`，通过 `payload.mode="LOOT_WINDOW"` 区分（旧协议）。

2. HTTP `/view`
- 用于拉取玩家当前快照（时间、状态、允许动作、倒计时、loot_window）。
- 调用时机较多：初始化、`ROUND_STARTED` 自动触发、动作提交后、部分控制操作后、手动刷新。

3. HTTP `/history`
- **不是固定轮询**，是“操作/事件触发式增量拉取”。
- 主要在初始化、动作提交后、`start/tick/reset/leave` 后、手动点击时调用。
- 当前不仅用于“历史展示”，还参与驱动部分 UI 状态更新。

## 2.2 业务流程现状（攻击/战利品）

1. `ATTACK` 在回合结算阶段判定。
2. 若进入战利品窗口，服务端推送 `ROUND_STARTED(mode=LOOT_WINDOW)`（旧协议）。
3. 胜者提交 `GET/TOSS` 后再触发 `ROUND_SETTLED`，再推进下一回合。
4. 败者侧感知较弱（“等待对方处理、不可关闭弹框、结果可见”语义不完整）。

## 2.3 当前问题

1. 事件语义混用  
`ROUND_STARTED` 同时表示“新回合开始”和“战利品子流程开始”，可读性与维护性差。

2. `/view` 触发点较多  
即使 REST/DEAD 等低收益场景，也可能随回合刷新触发请求。

3. `/history` 职责偏重  
既做日志又做状态驱动，职责边界不清晰。

4. 战利品双边体验不足  
败者等待态与结果可见性未形成明确协议保证。

## 2.4 当前实施状态（已完成）

1. 协议切换已完成
- 新事件已生效：`LOOT_WINDOW_STARTED`、`LOOT_WINDOW_RESOLVED`。
- `ROUND_STARTED` 仅用于新回合开始，并携带 `round_timer`。
- 旧 `ROUND_STARTED(mode=LOOT_WINDOW)` 协议已移除，不做兼容。

2. 接口返回口径已收敛
- `POST /rooms/{room_id}/actions` 在战利品分支返回中不再包含 `mode`。
- `POST /rooms/{room_id}/tick-ai` 返回中不再包含 `mode`。

## 3. 优化总览（To-Be）

四组工作并行推进，但建议按顺序落地：

1. 协议与事件模型重构（先定义）  
2. 后端流程改造（按新事件语义实现）  
3. 前端通信优化（按新协议收敛调用）  
4. 测试与验收（覆盖回归与性能）

---

## 4. 工作组A：协议与事件模型重构

## 4.1 改造目标

1. 解耦事件语义
- 新增 `LOOT_WINDOW_STARTED`
- 新增 `LOOT_WINDOW_RESOLVED`
- `ROUND_STARTED` 仅表示“新回合开始”

2. 补齐回合计时字段
- 在 `ROUND_STARTED.payload` 增加 `round_timer`
  - `opened_at`
  - `deadline_at`
  - `timeout_sec`

3. 明确战利品双边可见性
- `LOOT_WINDOW_STARTED`：双方都收到
- `LOOT_WINDOW_RESOLVED`：双方都收到，包含 `choice` 和 `obtained`

## 4.2 协议示例（建议）

1. `ROUND_STARTED`
```json
{
  "event_type": "ROUND_STARTED",
  "day": 2,
  "phase": "DAY",
  "round": 4,
  "payload": {
    "message": "day=2 phase=DAY round=4 choose action",
    "round_timer": {
      "opened_at": "2026-03-13T10:00:00Z",
      "deadline_at": "2026-03-13T10:01:30Z",
      "timeout_sec": 90
    }
  }
}
```

2. `LOOT_WINDOW_STARTED`
```json
{
  "event_type": "LOOT_WINDOW_STARTED",
  "day": 2,
  "phase": "DAY",
  "round": 4,
  "payload": {
    "winner_player_id": "A",
    "loser_player_id": "B",
    "expires_at": "2026-03-13T10:01:10Z"
  }
}
```

3. `LOOT_WINDOW_RESOLVED`
```json
{
  "event_type": "LOOT_WINDOW_RESOLVED",
  "day": 2,
  "phase": "DAY",
  "round": 4,
  "payload": {
    "winner_player_id": "A",
    "loser_player_id": "B",
    "choice": "GET",
    "obtained": {
      "bread": 1
    }
  }
}
```

## 5. 工作组B：后端流程改造

## 5.1 目标流程

攻击触发战利品时，回合内顺序固定为：

1. `ROUND_SETTLED`（核心动作结算）
2. `LOOT_WINDOW_STARTED`（双方进入子流程）
3. 胜者 `GET/TOSS` 或超时默认 `TOSS`
4. `LOOT_WINDOW_RESOLVED`（双方确认可见）
5. `ROUND_STARTED`（下一回合）

注：战利品子流程期间，回合号保持不变，结束后再进入下一回合。

## 5.2 行为要求

1. 子流程锁定
- 战利品窗口开启期间，双方普通动作不可提交。

2. 超时收敛
- 胜者超时未选，自动 `TOSS` 并广播 `LOOT_WINDOW_RESOLVED`。

3. 历史一致性
- `server_seq` 连续递增，`history` 可完整回放新事件。

## 5.3 代码改造点（建议）

- `src/api/constants.py`：新增事件常量
- `src/api/app.py`：新增 `publish_loot_window_started/resolved` 并重排触发顺序
- `src/application/match_service.py`：明确“同回合子流程”状态推进
- `src/api/payload_validation.py` + `docs/v1/schemas/`：新增/更新 schema

## 6. 工作组C：前端通信优化

## 6.1 现状到目标

1. `ROUND_STARTED` 到达后
- 现状：常通过 `/view` 拉取倒计时基准
- 目标：优先使用 WS 里的 `round_timer` 更新倒计时与顶部时间

2. `/view` 自动刷新
- 现状：回合触发较多场景都会调用
- 目标：保留“回合触发 + 条件跳过”
  - 默认新回合可拉一次
  - 若 `phase_ended=true` 且 `day/phase` 未变化，可跳过
  - 若 `alive=false` 且 `day/phase` 未变化，可跳过
  - 动作提交后、手动刷新、关键状态变化仍拉

3. `/history` 使用
- 现状：操作触发式补偿 + 参与部分状态驱动
- 目标：收敛为“关键补偿 + 历史展示”
  - 保留关键操作后补偿拉取：`submit/start/tick/reset/leave`
  - 历史 tab 可见时触发展示拉取
  - 减少以 history 驱动主视图状态

## 6.2 战利品弹窗体验（双边）

1. 胜者（A）
- 弹窗提示战斗胜利，可选 `GET/TOSS`。

2. 败者（B）
- 弹窗提示“等待对方缴获”，不可关闭。
- 收到 `LOOT_WINDOW_RESOLVED` 后显示“对方拿走了什么”。

3. 关闭规则
- 双方弹窗仅在 `LOOT_WINDOW_RESOLVED` 后关闭或变为可关闭。

## 7. 工作组D：测试与验收

## 7.1 功能验收

1. 事件语义
- `ROUND_STARTED` 不再承担 loot 含义。
- `LOOT_WINDOW_STARTED/RESOLVED` 双方都收到。

2. 战利品流程
- A 成功攻击 B 后，A 可选、B 等待不可关。
- A 选择后，B 可见 `obtained` 明细。

3. 回合推进
- loot 子流程期间回合号不变；结束后才进入下一回合。

## 7.2 稳定性验收

1. 断线恢复
- 重连后 `history` 能补齐 loot 相关事件且顺序正确。

2. 请求量优化
- 比对改造前后 `/view` 请求次数，REST/DEAD 场景明显下降。

3. UI 一致性
- 顶部时间、倒计时、可操作状态与服务端一致。

---

## 8. 分阶段落地计划

## Phase 1（协议先行）

- 定义新事件与 schema
- 后端提供兼容期（旧字段保留，新字段并行）

## Phase 2（后端切换）

- 上线 `LOOT_WINDOW_STARTED/RESOLVED`
- 调整回合推进顺序与超时处理

## Phase 3（前端切换）

- 前端消费新事件
- 倒计时改 WS 驱动
- 收敛 `/view` 和 `/history` 触发策略

## Phase 4（压测与回归）

- 场景回归（普通回合、attack->loot、超时、断线）
- 观测请求量、错误率与状态一致性

---

## 9. 风险与回滚

1. 风险
- 前后端事件协议不同步导致 UI 无法识别
- 子流程状态机改造引入回合卡死

2. 缓解
- 协议切换窗口：前后端同步发布，避免事件名不一致
- 增加服务端保护：loot 超时强制收敛

3. 回滚
- 保留旧事件分支开关；异常时切回旧推送路径

---

## 10. 结论

本优化并非“新增更多拉取”，而是：

- 用更清晰的 WS 事件语义减少歧义
- 用更精准的条件降低 `/view` 请求
- 保留 `/history` 的补偿价值但收敛其状态驱动责任
- 将攻击-战利品体验改造成双方可感知、可解释、可恢复的一致流程
