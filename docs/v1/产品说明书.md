## 3. 产品背景

在末日生存题材中，玩家核心体验来自“有限资源 + 高风险决策 + 信息不完全”。本产品通过同局（真人+AI）回合制机制，强调探索、争夺与生存选择，形成强策略对抗与可复盘的过程体验。

## 4. 产品定位与目标

### 4.1 产品定位

可配置房间上限的回合制末日生存策略游戏。真人与 AI 同局对抗，唯一目标是尽可能活得更久。

### 4.2 核心目标

- 构建“资源稀缺 + 信息滞后 + 冲突风险”三重压力下的策略博弈。
- 让每个动作都可量化地影响生存概率。
- 通过可配置终局模式适配不同对局节奏。

### 4.3 成功标准

- 玩家能清晰理解并使用 6 类核心动作完成生存闭环。
- 对局过程可复盘，胜负原因可解释。
- 规则在多角色并发提交下保持确定性与一致性。

## 5. 目标用户与使用场景

### 5.1 目标用户

- 偏策略推演的生存游戏玩家
- 喜欢“短回合、高决策密度”的对抗玩家
- 对 AI 对局有兴趣的混合对战用户

### 5.1 关系人定义（V1）

- 房主（真人）：创建房间、配置参数、发起开局。
- 玩家（真人/AI）：参与对局并提交动作。
- 结算方（系统）：服务端权威结算与裁决执行方。
- 关系说明：房主可以同时是玩家；结算方不属于玩家身份。

### 5.2 典型场景

- 房主创建房间后快速开局，真人不足可由 AI 自动补齐（受上限约束，不强制补满）。
- 玩家在白天高频探索与收集，夜晚做风险控制与休整。
- 围绕关键建筑与物资形成信息博弈和冲突选择。

## 7. 核心玩法概览

### 7.1 对局结构

- 一天由两个阶段组成：白天 -> 夜晚。
- 阶段内由多轮动作组成。
- 每轮每名存活角色最多提交 1 个核心动作。
- 攻击判定后会进入额外的战利品窗口动作（GET/TOSS）。

### 7.2 动作体系

核心动作：`MOVE / EXPLORE / USE / TAKE / REST / ATTACK`

后续动作（仅攻击后触发）：`GET / TOSS`

### 7.3 生存核心

- 资源不可再生，争夺具有零和性。
- 信息以个人认知为中心，快照可能失真。
- 冲突可带来收益，但会放大状态风险。

## 8. 系统规则说明

### 8.1 房间与匹配

- 单房间总人数上限由配置控制（默认 6）。
- 真人不足时可自动补齐 AI（默认最多 5），不强制补满房间。
- 房主创建房间时必须指定终局模式：
  - `ALL_DEAD`
  - `HUMAN_ALL_DEAD`

### 8.2 地图与区域

地图固定 9x9：

```text
[
  ["Q","Q","Q","Q","Q","Q","Q","Q","Q"],
  ["Q","X","W","X","X","J","X","J","Q"],
  ["Q","J","X","B","B","X","B","X","Q"],
  ["Q","X","S","M","X","W","X","J","Q"],
  ["Q","W","X","J","X","X","W","X","Q"],
  ["Q","B","X","X","X","W","J","J","Q"],
  ["Q","X","J","X","X","M","S","W","Q"],
  ["Q","X","X","X","J","B","X","B","Q"],
  ["Q","Q","Q","Q","Q","Q","Q","Q","Q"]
]
```

区域定义：
- `Q`：昼夜进入立即死亡
- `X`：白天进入立即死亡；夜晚按曝光计算存活率
- `J/B/S/W/M`：安全建筑区域

### 8.3 状态与初始配置

状态类型：水分、饱食、曝光

初始与上限（V1）：
- 水分：初始 `100`，上限 `100`
- 饱食：初始 `100`，上限 `100`
- 曝光：初始 `0`，上限 `100`
- 初始背包：`瓶装水 x1`、`面包 x1`

死亡条件：
- 水分 `<=0` 或饱食 `<=0`
- 白天在 `Q/X`
- 夜晚在 `X` 且判定失败

### 8.4 物资与库存

库存规则：
- 建筑库存固定且不可再生
- 探索后可知库存，拿取后即时扣减

建筑初始库存：
- `J`：面包 x3、瓶装水 x3
- `B`：压缩饼干 x4
- `S`：罐头 x5、桶装水 x5
- `W`：清水 x6
- `M`：无可拿取物资

物品效果：
- 面包：饱食 +10
- 瓶装水：水分 +10
- 压缩饼干：饱食 +20
- 罐头：饱食 +20
- 桶装水：水分 +20
- 清水：水分 +15

### 8.5 信息与记忆机制

- 玩家初到建筑时：物资/角色信息默认未知。
- `EXPLORE` 后获得物资快照与角色快照。
- 角色快照是单向可见。
- 快照长期保留，不做硬失效；但可能失真（STALE）。
- 玩家仅在本人参与动作后刷新该建筑认知：
  - 会刷新：`EXPLORE`、`TAKE`、`ATTACK`、`GET/TOSS`
  - 不刷新：`REST`、`USE`

### 8.6 消耗规则（V1）

阶段固定消耗（每阶段仅一次）：
- 水分 `-1`、饱食 `-1`、曝光 `0`

动作额外消耗：
- `MOVE`：水分 `-2`、饱食 `-1`、曝光 `+2`
- `EXPLORE`：水分 `-1`、饱食 `-1`、曝光 `+1`
- `USE`：水分 `0`、饱食 `0`、曝光 `0`（仅结算物品效果）
- `TAKE`：水分 `-1`、饱食 `0`、曝光 `+1`
- `REST`：水分 `0`、饱食 `0`、曝光 `-3`
- `ATTACK`：水分 `-2`、饱食 `-2`、曝光 `+3`
- `GET`：水分 `0`、饱食 `0`、曝光 `0`
- `TOSS`：水分 `0`、饱食 `0`、曝光 `0`

状态边界处理：
- 每轮状态汇总后先执行 `clamp(0,100)`，再执行死亡判定。

### 8.7 夜晚 X 区存活规则（V1）

- 存活函数：`P_survive(E) = clamp(0.97 - 0.008 * E, 0.03, 0.97)`
- `E` 为曝光值。
- 每轮生死判定阶段，对每个处于 `X` 的存活角色执行一次独立判定。

### 8.8 冲突与战利品

攻击值：
- `AttackScore = Base + StateMod + InfoMod + Rand`
- `Base = 10`
- `InfoMod`：单边发现 `+2`，互相发现等效抵消
- `Rand ∈ {-1, 0, +1}`
- `StateMod = ResourceMod + ExposureMod`

胜负阈值：
- `Δ = 发起方AttackScore - 被攻击方AttackScore`
- `Δ >= 2` 发起方胜
- `Δ <= -2` 被攻击方胜
- `-1 <= Δ <= 1` 僵持

战利品窗口：
- 攻击判定后胜者二选一：`GET(items)` 或 `TOSS`
- `GET` 最多拿 3 件
- 超时默认 `TOSS`

### 8.9 结算顺序与并发冲突

同轮结算顺序：
- 先按动作提交时间排序
- 同时间按进入房间顺序排序（先进入先结算）

共享资源冲突：
- 先结算先扣库存，后结算可能拿不到或部分拿到。

死亡掉落：
- 死亡角色背包剩余物资掉落到当前建筑库存。

### 8.10 玩家信息面板与通知机制

固定信息（所有在局玩家，包含房主）：
- 我是谁（玩家标识）
- 我所在房间
- 我的位置坐标
- 当前时间态（第几天、白天/夜晚）
- 当前建筑披露信息状态（未探索/上次探索快照）
- 当前建筑快照内容（角色信息、物资信息）
- 我的状态（水分、饱食、曝光）
- 我的背包物品

操作历史：
- 历史消息仅保留最近 100 条，滚动刷新。

通知事件（V1）：
- 游戏开始：广播“房间 N 人（含 AI）进入游戏”
- 回合开始：广播“第 X 天、白/夜阶段、请玩家选择操作”
- 动作提交：仅向提交者回执“已选择操作，等待结算”
- 结算完成：向所有人广播“本轮结算完成”，并下发各自私有结算结果

超时配置（V1）：
- 核心动作提交超时：`round_action_timeout_sec = 90`
  - 真人玩家超时默认 `REST`
  - AI 由系统自动决策并提交动作
- 战利品窗口超时：`loot_window_timeout_sec = 60`
  - 超时默认 `TOSS`

结算结果展示原则：
- 仅展示与本人相关的结算详情，不展示他人私有细节
- 各动作应输出可解释结果（消耗、增益、目标、结果）
  - 探索：消耗与探索发现
  - 移动：消耗与新坐标
  - 使用：消耗与状态恢复
  - 休息：状态变化与阶段结束信息
  - 获取：获得物品清单
  - 攻击：目标、胜负、状态变化、战利品窗口结果

## 9. 关键业务流程

### 9.1 标准回合流程

1. 存活角色提交动作（或超时自动 `REST`）
2. 系统统一结算（按顺序队列）
3. 若有攻击，进入战利品窗口（GET/TOSS）
4. 状态汇总并执行边界处理
5. 执行生死判定与死亡掉落
6. 满足阶段结束条件则切阶段，否则继续下一轮

全员 `REST` 特殊推进口径（V1）：
- 本轮 `REST` 仍按常规动作结算（含 `REST` 动作效果）
- 本轮结束后直接推进到下一阶段，不再开启同阶段下一轮

### 9.2 单人阶段流程

当阶段内只剩 1 名存活角色：
- 该角色可继续在同一阶段执行多轮动作
- 直到该角色 `REST` 或死亡才结束该阶段

## 10. AI 行为策略（产品口径）

- 目标：最大化生存概率
- 白天偏探索/移动/收集
- 夜晚偏休息/隐蔽
- 资源不足时优先补给
- 仅在明显优势且必要时攻击
- 策略接口统一：`choose_action(obs, action_mask) -> action`
- V1 策略实现：`RuleBot + LLMPolicy`
- `LLMPolicy` 采用两层机制（发现层 + 意图层），失败回退 `RuleBot`
- 策略分层：场景模板层（按昼夜/风险/窗口路由）+ 两层决策（发现层/意图层）
- 模板可配置化：支持后续新增模板而不改核心结算流程

## 11. 终局与战绩输出

终局触发：
- `ALL_DEAD`：全角色死亡
- `HUMAN_ALL_DEAD`：真人全灭

终局输出：
- 角色统计：存活轮次、物资获取总量、死亡原因
- 真人战绩：最后存活真人、真人存活时长、击杀/被击杀
- 排名：按存活轮次降序

## 12. 关键指标建议（运营/分析）

- 人均存活天数
- 平均每局轮次
- 行为分布（MOVE/EXPLORE/TAKE/REST/ATTACK）
- 攻击触发率与反杀率
- 夜晚 X 区死亡占比
- 终局模式下真人存活率

## 13. 风险与控制

- 风险：规则复杂导致理解成本高
  - 控制：在产品内提供简明规则卡与回合结算日志
- 风险：并发结算争议
  - 控制：公开结算顺序原则（提交时间 + 入房顺序）
- 风险：数值平衡漂移
  - 控制：按版本管理参数并保留灰度调参能力
